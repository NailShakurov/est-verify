# EST — генератор именных сертификатов с подписанным QR

CLI-скрипт: на вход таблица выпускников (`.xlsx`) и пустой PDF-шаблон, на выходе —
папка готовых именных PDF-сертификатов. На каждый сертификат наносится ФИО и
персональный QR-код. Внутри QR — данные сертификата (номер, ФИО, курс, дата),
подписанные приватным ключом Ed25519. Подлинность сертификата проверяется по QR
(отдельный веб-верификатор, вне рамок этого README) или локально через
`verify_token.py`.

Корень доверия: цифровая подпись + напечатанный домен проверки (`est.com.kz`) рядом
с QR.

## 1. Установка

Требуется Python 3.11+.

```bash
cd generator
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Зависимости запинены в `generator/requirements.txt`: `PyMuPDF`, `segno`, `PyNaCl`,
`openpyxl`, `PyYAML`, `pytest`.

## 2. Шрифт для кириллицы

ФИО на сертификате рисуется кириллическим TTF-шрифтом. В репозитории уже
забандлен `generator/assets/DejaVuSans.ttf` — коммитить и скачивать заново не
нужно. Если файл вдруг отсутствует, его можно получить одним из способов:

```bash
curl -L -o generator/assets/DejaVuSans.ttf \
  https://github.com/dejavu-fonts/dejavu-fonts/raw/version_2_37/ttf/DejaVuSans.ttf
# или скопировать из системных шрифтов, если есть:
cp /usr/share/fonts/truetype/dejavu/DejaVuSans.ttf generator/assets/DejaVuSans.ttf
```

Путь к шрифту задаётся в `config.yaml` ключом `font_file`.

## 3. Генерация ключей подписи

Сертификаты подписываются парой ключей Ed25519 (хранятся как base64-текст).

```bash
cd generator
python generate.py --genkey
```

Создаст `./keys/ed25519_private.key` (права `600`) и `./keys/ed25519_public.key`
согласно путям в `config.yaml` (`keys.private` / `keys.public`). Если ключи уже
существуют, команда откажется их перезаписывать (`FileExistsError`) — приватный
ключ нужно удалять вручную и осознанно, если действительно нужна новая пара
(старые сертификаты со старым ключом перестанут проходить проверку).

**ВАЖНО:** приватный ключ (`ed25519_private.key`) никогда не покидает локальную
машину — не коммитится, не логируется, не копируется на сервер заказчика.
Публичный ключ (`ed25519_public.key`) безопасен для распространения — он
передаётся в веб-верификатор и используется скриптом `verify_token.py`.

## 4. Подготовка `input/`

Каталог `generator/input/` не коммитится (в нём — персональные данные заказчика).
В него нужно положить два файла заказчика:

- `input/template.pdf` — пустой одностраничный PDF-шаблон сертификата;
- `input/students.xlsx` — таблица выпускников.

Пути задаются в `config.yaml` (`template_pdf`, `table_path`).

### Маппинг колонок

Заголовки колонок в таблице заказчика могут называться как угодно — соответствие
нашим трём обязательным полям задаётся в `config.yaml`, секция `columns`. Текущая
конфигурация:

```yaml
columns:
  fio: "Фамилия Имя"
  date: "Дата"
  number: "Номер сертификата"
```

То есть в `students.xlsx` в первой строке (заголовки) должны быть колонки с ровно
такими названиями: `Фамилия Имя`, `Дата`, `Номер сертификата`. Остальные колонки
таблицы (например «№», «Кем выдан») не читаются — их можно оставить как есть.
Дата принимается либо как дата/datetime-ячейка Excel, либо как строка в одном из
форматов `YYYY-MM-DD`, `DD.MM.YYYY`, `DD/MM/YYYY`; в payload она всегда попадает в
нормализованном виде `YYYY-MM-DD`.

Перед генерацией стоит проверить таблицу без побочных эффектов:

```bash
python generate.py --check
```

Выведет `OK: N строк готовы к генерации.` либо список всех найденных ошибок
(пустые поля, дубликаты номеров сертификатов, нераспознанные даты, запрещённый
символ-разделитель `U+001F` в данных) — генерация при ошибках не запускается.

## 5. Выверка координат (`--preview`)

Координаты ФИО, QR и лейбла домена задаются в `config.yaml`, секция `placement`
(в точках PDF, начало координат — верхний левый угол страницы):

```yaml
placement:
  fio:  { x: 300, y: 250, fontsize: 18, fio_min_fontsize: 10, max_width: 260, color: "#000000" }
  qr:   { x: 450, y: 600, size: 90 }              # 90 pt ≈ 3.2 см (>= 3 см)
  verify_label: { x: 450, y: 695, fontsize: 8, font: "helv" }
```

Чтобы подобрать координаты под конкретный шаблон, не запуская весь батч,
используется `--preview`:

```bash
python generate.py --preview        # 1 сертификат (по умолчанию)
python generate.py --preview 3      # первые 3
```

Откройте получившийся PDF из `output/`, скорректируйте `x`/`y`/`size`/`fontsize`
в `config.yaml` и повторите. ФИО центрируется по точке `x` и автоматически
уменьшает размер шрифта (до `fio_min_fontsize`), если строка не помещается в
`max_width`. Требование к QR — сторона не меньше 3 см (в конфиге — 90pt с
запасом).

## 6. Запуск полного батча

```bash
python generate.py
```

Сгенерирует сертификаты для всех валидных строк таблицы в `output/<Номер>_<ФИО>.pdf`
(имя файла санитизируется: пробелы → `_`, убираются символы `/ \ : * ? " < > |`,
кириллица сохраняется). В конце выводится максимальная версия QR среди всех
сертификатов; если она больше 10 — печатается предупреждение проверить
читаемость QR при печати/сканировании (сложный QR с длинными ФИО/названием курса
может стать нечитаемым).

### Компактный режим

Если название курса длинное и приводит к слишком плотному QR, можно записывать в
QR короткий код курса вместо полного названия:

```bash
python generate.py --compact
```

Код берётся из `config.yaml` (`course_code`, сейчас `"E"`). Режим также можно
включить по умолчанию для всех запусков, поставив `compact_course: true` в
конфиге — тогда `--compact` необязателен (флаг и `compact_course: true`
эквивалентны, флаг работает даже если в конфиге `false`).

Полный список режимов CLI: `--genkey`, `--check`, `--preview [N]`, `--compact`,
`--config <path>` (по умолчанию `config.yaml`), без флагов — полный батч.

## 7. Приёмочная проверка (`verify_token.py`)

После генерации партии стоит вручную проверить хотя бы один сертификат: открыть
его QR (или взять URL напрямую) и прогнать через `verify_token.py` —
самостоятельный скрипт, который декодирует payload из токена и проверяет подпись
публичным ключом:

```bash
python verify_token.py "https://est.com.kz/#<токен>" --public ./keys/ed25519_public.key
```

Скрипт принимает как полный URL (данные после `#`), так и голый токен. Выводит 4
поля (номер сертификата, ФИО, курс, дата выдачи) и статус:

- `СТАТУС: ПОДЛИННЫЙ (подпись верна)` — код возврата `0`;
- `СТАТУС: ПОДПИСЬ НЕДЕЙСТВИТЕЛЬНА` (в stderr) — код возврата `2`, подпись не
  сошлась с публичным ключом.

Рекомендуется как минимум распечатать 2-3 сертификата и отсканировать QR
телефоном, чтобы убедиться, что QR версии ~10-11 всё ещё читается печатью.

## 8. Безопасность

- **Приватный ключ** (`keys/ed25519_private.key`) — только на локальной машине,
  где идёт генерация. Не коммитится, не логируется, не копируется на сервер
  заказчика. `--genkey` создаёт его сразу с правами `600`.
- **Персональные данные** (`input/students.xlsx`, `input/template.pdf` с реальными
  данными, `output/*.pdf`) — тоже не коммитятся.
- `.gitignore` в корне репозитория уже исключает `keys/`, `*.key`, `input/`,
  `output/` (плюс служебные `__pycache__/`, `.venv/`, `.pytest_cache/`).
- **Ручная заливка на сервер заказчика** (например, `scp`/`rsync` итоговых PDF)
  должна явно исключать `keys/`, `input/`, `output/` — заливать нужно только
  итоговые именные PDF, не рабочие каталоги генератора. Пример:
  ```bash
  rsync -av --exclude 'keys/' --exclude 'input/' --exclude 'output/' \
    ./output/ user@server:/path/to/certs/
  ```
  (то есть `output/*.pdf` можно и нужно заливать, но каталоги `keys/` и `input/`
  генератора на сервер заказчика не попадают ни при каких условиях).
- **Публичный ключ** (`keys/ed25519_public.key`) безопасен для распространения —
  он идёт в веб-верификатор и используется для приёмочной проверки.
- Данные QR лежат во фрагменте URL (`#...`), а не в query-параметрах — фрагмент не
  уходит на сервер при переходе по ссылке (важно с точки зрения 152-ФЗ о ПДн).
